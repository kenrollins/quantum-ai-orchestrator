# quantum-ai-orchestrator — Implementation Plan

## 1. Context

**`quantum-ai-orchestrator`** is an **AI-orchestrated control plane for hybrid quantum-classical workloads** — a public, open-source demonstration of the operating model that connects today's GPU-based quantum-inspired and quantum-simulator backends to tomorrow's QPUs through a single AI-driven pipeline. Built specifically as an AI + quantum talk track and demo for **Dell Federal**.

**What it does:** takes a natural-language ask, decomposes it via an LLM into a problem graph, dispatches each leaf in parallel across a portfolio of GPU-backed solvers (classical, quantum-inspired annealing, GPU statevector simulation, AI quantum-error-correction decoders), races them, picks the winner, reassembles the answer, and records every decision in a Postgres-backed auditable provenance log. Substrate today is GPUs; the pipeline is designed so QPU backends slot in unchanged when they arrive.

**What it isn't:** not a simulator (it *uses* simulators); not a runtime substrate (CUDA-Q already is one; we sit above it); not a quantum-control startup (Conductor Quantum and others operate at the QPU calibration layer; we operate at the workload-orchestration layer).

**Strategic anchor:** NVIDIA's open **Ising** model family (released 2026-04-14 on World Quantum Day) is the federal hero ingredient. Specifically the **Ising Decoding** model (3D-CNN, open weights, 2.5× faster and 3× more accurate than traditional decoders). We integrate it as a backend and race it live against **PyMatching** (the standard classical min-weight-matching decoder) on simulated surface-code syndromes generated by **Stim**. The "AI is making quantum reliable" demo plays out in real time, on Dell hardware, with no QPU required.

**Talk track:** *"Quantum needs AI to be useful. NVIDIA shipped that AI on April 14. Dell ships the on-prem iron it runs on. Your federal QPU pilot needs both — and you can stand the AI side up before the QPU arrives."*

**Hero use-case ordering:** lead with `qec_decode` (Ising-vs-PyMatching live race — *AI fixes quantum today*), follow with `mission_assignment` (orchestration thesis — *AI builds the QUBO, four backends race in parallel*). Phase 2 adds `routing`. Phase 3 adds `portfolio`. Portfolio is third because financial framing reads as Wall Street to some federal audiences.

**Public from day one:** Apache 2.0 license (matches NVIDIA Ising, Stim, PyMatching, CUDA-Q). README is part of Phase 0. No `dell-` prefix in the repo name (README does the Dell positioning). No personal-flavored framing in the README opener — lead with what the project *is*, not who built it.

## 2. Architectural Shape

This is a **pipeline**, not a retry loop. A natural-language ask passes through six stages exactly once: Decomposer → Formulator → Dispatcher → Backends (in parallel) → Evaluator → Reassembler. A backend losing the race is not a failure — it is data. Learning happens between asks (via the Strategist), not within.

Key constraints:
- **No rollback semantics.** Each ask is one pass; there is no mid-task recovery.
- **Postgres for provenance.** Bi-temporal `valid_from / valid_to` columns; audit queries are SQL. No graph database in Phase 1–2.
- **Scored outcomes, not categorized failures.** We learn positive preferences, not bans.
- **Skills are thin.** Each skill provides three Python functions and some markdown — no protocol inheritance.

## 5. Pipeline Architecture

**No Ralph loop. No five-Protocol inheritance.** A clean pipeline.

```
NL ask
  └─ Decomposer (Gemma 4 31B Dense, JSON-output)
       └─ problem-graph DAG
            └─ Formulator (per-skill, dispatches by problem_class)
                 └─ backend-ready inputs (QUBO matrix / Ising J,h / syndrome tensor)
                      └─ Dispatcher (reads config/backends.yaml + Postgres lessons table)
                           ├─ Backend A (parallel)  ─┐
                           ├─ Backend B (parallel)  ─┤── Evaluator (per-backend; quality + wall_time)
                           └─ Backend C (parallel)  ─┘
                                ├─ Reassembler (walks DAG bottom-up)
                                │    └─ final answer to user
                                └─ Strategist (optional, between asks)
                                     └─ updates Postgres lessons table
                                                    │
                                                    └─ All decisions appended to Postgres provenance tables
```

**Pipeline stage modules** (in `orchestrator/pipeline/`):

- **`decomposer.py`** — single async function `decompose(ask: str) -> ProblemGraph`. Calls Gemma 4 via Ollama with a JSON-schema-constrained prompt. Output is a typed DAG: `Problem(class, fingerprint, parent_id, params)`. Skill-specific decomposer prompts loaded from `skills/<name>/prompts/decomposer.md`.
- **`formulator.py`** — pure router: dispatches each leaf to the skill's formulator based on `problem_class`. Skill formulators live at `skills/<name>/formulator.py`. Output: `BackendInput` (typed by problem class).
- **`dispatcher.py`** — reads `config/backends.yaml` + Postgres `lessons` table. Returns `list[BackendChoice]` per leaf. Default policy: pick top-K backends by learned preference; race them in parallel.
- **`backends/*.py`** — each backend implements `run(input: BackendInput, gpu_lane: int | None) -> Solution`. GPU-lane assignment by dispatcher (GPU0 / GPU1 / CPU).
- **`evaluator.py`** — pure: takes `(BackendInput, Solution)`, returns `Grade(quality, wall_time, energy_or_LER)`. For optimization: quality vs OR-Tools/LP lower bound. For QEC: LER vs uncorrected baseline. Skill-specific evaluators live at `skills/<name>/evaluator.py`.
- **`reassembler.py`** — pure: walks problem-graph bottom-up combining leaf solutions. Identity for single-leaf graphs. For decomposed problems, applies the skill's combine function.
- **`strategist.py`** — *optional, runs between asks*. Reads recent runs from Postgres, distills `(problem_fingerprint, backend, outcome)` triples into updates to the `lessons` table. Implementation: a periodic SQL job + LLM-narration of newly-learned preferences. Phase 2.

**Cross-cutting commitments locked day-1** (so adding skills 3 and 4 is folder-only):

1. **Backend registry** (`config/backends.yaml`) — schema accommodates all 7 backend types day-1. Each entry: `{name, class, applicable_problem_classes, gpu_required, footprint_gb, latency_target_ms}`. Phase 1 ships entries for `classical_ortools`, `neal`, `cudaq_qaoa`, `ising_speed`, `ising_accuracy`, `pymatching`. Phase 2 adds `cutensornet`. Phase 3 adds `pennylane_var`.
2. **Problem-graph schema** — `Problem.problem_class` is a closed enum from day 1: `{qec_syndrome, qubo_assignment, qubo_routing, qubo_portfolio}`. Formulator and Evaluator dispatch on it.
3. **Skill manifest format** (`skill.yaml`) — locked schema: `{name, accepted_problem_classes, decomposer_prompt, formulator_module, evaluator_module, ui_panel_module, color_tokens, postgres_schema}`.
4. **Postgres schemas** — five schemas in the `quantum_ai_orchestrator` database day-1: `common` (runs, provenance), `qec_decode`, `mission_assignment`, `routing`, `portfolio`. Phase 2/3 schemas pre-created empty.
5. **Provenance tables** — `runs`, `problem_graphs`, `dispatches`, `outcomes`, `lessons` with bi-temporal `valid_from / valid_to` columns. Audit query: `SELECT lesson FROM lessons WHERE valid_from <= $T AND (valid_to IS NULL OR valid_to > $T) AND problem_class = $C`. Plain SQL.
6. **Dashboard panel registry** — front-end has a panel-loader pattern. Each skill ships its own React component; adding a skill = adding a panel module + registering it in `config/dashboard.yaml`.

## 7. Skills

Each skill is `/data/code/quantum-ai-orchestrator/skills/<name>/`. **Thinner contract than gemma-forge:** a skill provides:
- `skill.yaml` — manifest (locked schema in §5)
- `formulator.py` — function `formulate(problem: Problem) -> BackendInput`
- `evaluator.py` — function `evaluate(input: BackendInput, solution: Solution) -> Grade`
- `prompts/decomposer.md`, `prompts/strategist.md` (optional)
- `ui/<PanelName>.tsx` — React component for the skill's domain panel

No five-Protocol inheritance. No retry loops. Just three Python functions and some markdown.

### `qec_decode` (Phase 1 — federal hero)
**Pipeline:** distance ∈ {3, 5, 7}, depolarizing noise rate p, shots → Stim simulates noisy logical qubit and emits syndrome stream → three decoders race on the same stream: `ising_speed`, `ising_accuracy`, `pymatching` → Evaluator computes LER vs uncorrected baseline → Strategist (Phase 2) learns *"high-noise → accuracy variant; low-noise → speed; very-low-noise → PyMatching is fine"*.

**Demo:** live LER curve as p slides from 1e-4 to 1e-2; three lines diverge; the QEC Lab panel shows the surface-code lattice with syndrome heatmap and decoder-corrected overlay. ~60-second demo.

**Why hero:** AI literally making quantum reliable, using a model NVIDIA released 12 days ago, on Dell hardware, no QPU needed.

### `mission_assignment` (Phase 1 — orchestration thesis)
**Input:** M assets × T tasks with skill-fit matrix S, capacity constraints. **Formulation:** weighted assignment QUBO `max Σ S_{a,t} x_{a,t} − penalties`. **Demo data:** deliberately abstract (assets A1..A12, tasks T1..T8 — no customer attribution). **Sizes:** 12×8 = 96 binary vars QAOA, 24×16 neal, anything OR-Tools.

**Demo:** AI decomposes the ask, dispatches to four backends in parallel (`classical_ortools` CPU, `neal` CPU, `cudaq_qaoa` GPU0, `cudaq_qaoa` with different params on GPU1), evaluator scores them, Strategist (Phase 2) learns. Bipartite matching graph that morphs as backends propose alternatives. ~90-second demo.

### `routing` (Phase 2)
CVRP as QUBO via position-encoding with subtour-elimination penalties. Synthetic 15–20-stop tour. Animated route panel via deck.gl.

### `portfolio` (Phase 3)
Markowitz with cardinality penalty. yfinance S&P 100 subset. Efficient frontier panel.

## 11. Risks and Open Questions

1. **Containerized CUDA-Q + driver 535 forward-compat.** Verify in Phase 0; if it fails, fall back to `qiskit-aer-gpu` for Phase 1 QAOA and write an ADR documenting the gap.
2. **NVIDIA Ising weights license** — confirm redistribution terms before Phase 0 push. Default: bootstrap script pulls at first run; weights live at `/data/models/nvidia-ising/` (out of repo). ADR-0001 documents.
3. **Two-GPU racing fairness** — wall-time across backends must control for warm-up; add 1-iter warm-up before timed runs or the comparison reads as cooked.
4. **Gemma 4 31B q8 contention** — 33 GB model competing with sim VRAM. Default: pin Ollama to GPU0 (or share across both, accepting eviction); dispatch GPU-bound sim backends to whichever lane has VRAM headroom. ADR-0002 documents.
5. **Supabase tenancy isolation** — the workstation's supabase-db hosts other projects. Always operate scoped to the `quantum_ai_orchestrator` database; never `DROP` outside our schemas. Migrations are idempotent (`CREATE SCHEMA IF NOT EXISTS …`).
6. **Credential hygiene** — `/data/docker/supabase/.env` contains many secrets we don't need (Langfuse, JWT). Read only the keys we need (`POSTGRES_PASSWORD`); copy *only those* into project `.env`; never commit either. ADR-0001 documents.
7. **Conductor Quantum positioning** — they're a YC startup adjacent to us (AI for QPU calibration; NVIDIA Ising adopter). README differentiates clearly: *we are the workload orchestrator above CUDA-Q, not a calibration product*. Phase 0 README section.
8. **Stim + PyMatching API surface** — PyMatching 2.x changed APIs from 1.x; pin the version and integration-test in Phase 0.
9. **Ollama model-version drift** — `gemma4:31b-it-q8_0` tag is stable but not immutable. Pin via SHA in `tools/bootstrap_models.sh` for reproducibility.
10. **Dashboard SSE bandwidth** — `statevector_snapshot` and `syndrome_emitted` payloads must be downsampled or live demo lags. Phase 1 instrumentation includes per-event size logging.
11. **Public repo + Dell brand** — README must position carefully (Dell + NVIDIA narrative without using Dell trademarks in repo name or logo). Run journey entries through a brand check before push if any reference Dell directly.
12. **yfinance redistribution** — Phase 3 only. Ship a fetch script, not CSVs.
13. **Crumble hosting decision** — running Crumble from `algassert.com` leaks demo activity to a third-party server and won't work in any air-gapped or federal-sensitive environment. If we use it, we must self-host the open-source build (adds ~5 MB JS dep). ADR-0008 in Phase 1.
14. **D-Wave problem-inspector network behavior** — the inspector spawns a local web viewer; verify it doesn't phone home to D-Wave Leap. If it does, isolate via firewall rules or extract the SVG render directly. Phase 0 verification.

## 12. NVIDIA Ising Integration — Deep Dive

This is the differentiator. Treat as a first-class concern.

- **Source.** Open weights via HuggingFace Hub. Bootstrap via `tools/bootstrap_ising.sh` at first run; cache to `/data/models/nvidia-ising/` (gitignored).
- **Models for Phase 1**: Decoding speed variant (~30M params, 4 GB FP16) + accuracy variant (~120M params, 12 GB FP16). Both are 3D CNNs.
- **Pipeline:**
  1. `orchestrator/quantum/noise/surface_code.py` — Stim-driven syndrome generation for distances 3, 5, 7 under depolarizing noise.
  2. `orchestrator/quantum/backends/ising_decoder.py` — wraps Ising-Decoding inference (speed + accuracy as separate backend entries); returns correction + confidence + latency.
  3. `orchestrator/quantum/backends/pymatching_decoder.py` — PyMatching baseline backend.
  4. Skill `qec_decode/formulator.py` ties it together.
- **CUDA-Q QEC 0.6 alignment.** Mirror its API for the backend wrapper so a future swap to a real QPU connected via NVQLink is a config change, not a code change.
- **Strategist learning target (Phase 2).** Which decoder for which `(distance, p)` regime. Materialized as rows in `common.lessons`.
- **Honest demo framing.** No real QPU = no real-time decoding loop. The demo is **offline batch decoding of simulated syndromes**. State that in the dashboard footer.
